# Incremental Observability List Plan

## Scope

Add stateless incremental polling to all pollable observability list endpoints:

- `listTraces`
- `listLogs`
- `listScores`
- `listFeedback`
- `listMetrics`
- `listBranches`

This work should land for:

- ClickHouse v-next observability
- DuckDB observability
- internal in-memory observability store

## Goals

- return only new or changed data after a client-held cursor
- use ingestion order, never client event time, as the polling boundary
- tolerate late and out-of-order arrivals
- require no server-side polling state
- preserve one canonical list endpoint per resource
- keep the design compatible with a later SSE or streaming upgrade

## Non-Goals

- changing trace or signal display timestamps
- introducing subscriptions or server-held cursors
- solving cross-client merge policy in the frontend beyond deterministic upsert semantics
- changing OLAP aggregate and timeseries endpoints in this pass

## Recommendation

Do not add new `/updates` endpoints.

Extend the existing list endpoints with a second query mode:

- page mode: existing `page` / `perPage` / `orderBy`
- delta mode: `after?` / `limit`

These modes should be mutually exclusive. Requests that mix page pagination with delta polling should fail validation.

Also:

- missing `mode` should continue to behave like page mode for backward compatibility
- accept explicit `mode=page` as an alias for the default paged behavior
- require explicit `mode=delta` to enter incremental polling semantics
- page mode should return a `liveCursor` snapshot watermark for resuming with delta mode when the backend can provide one
- page-mode `liveCursor` should be optional in the response shape so legacy stores can continue returning paged lists without implementing delta bootstrap immediately
- do not add a separate history cursor in this pass
- keep existing history browsing semantics for now

## Shared Cursor Contract

Use one public cursor shape everywhere:

```ts
type LiveCursor = {
  ingestedAt: string; // ISO timestamp generated by the backend at write time
  tieBreaker: string; // stable secondary key for same-timestamp collisions
};
```

`LiveCursor` is the public API type. "Ingestion cursor" is still a useful internal term for the underlying ordering concept.

Rules:

- `ingestedAt` is backend-generated, not client-supplied
- `tieBreaker` must be stable and unique within the endpoint's backing row stream
- ordering is lexicographic on `(ingestedAt, tieBreaker)`
- delta reads are strictly after the cursor, never `>=`
- delta responses always return the highest returned cursor as `liveCursor`
- if the result set is empty, return the request cursor unchanged, or `null` if none was supplied
- `ingestedAt` is internal cursor infrastructure and should not be exposed as a normal field on returned observability objects
- the only public surface for ingestion ordering metadata in this pass is the cursor object itself

## Endpoint Contract

Each list endpoint keeps one path and supports two modes.

### Page Mode

Use when:

- `mode` is omitted
- `mode=page`

Current behavior, with one addition:

- response includes `liveCursor`, representing the filtered snapshot high-water mark at query time
- `liveCursor` is for future delta polling, not for paging older results

Example:

```ts
type PageListResponse<T> = {
  items: T[];
  pagination: {
    total: number;
    page: number;
    perPage: number;
    hasMore: boolean;
  };
  liveCursor?: LiveCursor | null;
};
```

Important:

- `liveCursor` must not be derived from only the visible page items
- it should represent the newest matching ingested row or activity in the full filtered snapshot as of query time
- this allows one page-mode request to both hydrate the UI and start correct incremental polling

### Delta Mode

Use:

```ts
type DeltaListArgs = {
  mode: 'delta';
  after?: LiveCursor;
  limit?: number;
};
```

Response:

```ts
type DeltaListResponse<T> = {
  items: T[];
  delta: {
    limit: number;
    hasMore: boolean;
  };
  liveCursor?: LiveCursor | null;
};
```

Notes:

- `mode=delta` is required for delta semantics
- when `after` is omitted in delta mode, start from "now" and return an empty item set plus the current `liveCursor`
- when `after` is omitted in delta mode:
  - if the filtered result set has any matching rows with non-`NULL` `ingestedAt`, mint `liveCursor` from the max matching `(ingestedAt, tieBreaker)`
  - otherwise mint a synthetic "now" `liveCursor`
  - in both cases return `items: []`
- `limit` is not page size; it is the max number of updates returned in this poll window
- `hasMore=true` means the client should poll again immediately using the returned `liveCursor`
- filters must behave identically in page mode and delta mode
- delta requests should accept the same filters as page mode and only return newly ingested matching rows or snapshots
- a `liveCursor` is only valid relative to the filter set it was created from
- changing filters should be treated as a new query that obtains a new `liveCursor`
- do not add `filterHash` in v1; document the same-filter-set requirement instead
- HTTP transport for `after` should align with existing server query parsing conventions for complex params
- prefer structured query params such as `after[ingestedAt]=...&after[tieBreaker]=...`
- JSON-string form for `after` is also acceptable if it falls out naturally from the existing query wrapper path
- do not introduce a new opaque encoded cursor transport format in v1

## Per-Signal Semantics

### Traces

Traces are not simple append-only list rows from the UIâ€™s perspective.

Delta semantics for traces should be:

- return traces whose latest activity moved past the cursor
- return the current root-trace snapshot for each changed trace
- frontend merges by `traceId`

This is necessary because:

- a late child span can change `hasChildError`
- a late child span can change trace completeness or derived trace metadata
- ordering by root `startedAt` or `endedAt` is not safe for polling correctness

Trace delta ordering should therefore be based on trace activity ingestion order, not root span time.

Recommended trace activity definition:

- latest ingested row for any span in the trace

### Logs

Logs are row-level append-only events.

Delta semantics:

- return ingested log rows strictly after the cursor
- frontend appends or de-dupes by `logId`

### Scores

Scores are row-level append-only events.

Delta semantics:

- return ingested score rows strictly after the cursor
- frontend appends or de-dupes by `scoreId`

### Feedback

Feedback is row-level append-only.

Delta semantics:

- return ingested feedback rows strictly after the cursor
- frontend appends or de-dupes by `feedbackId`

### Metrics

Metrics are row-level append-only.

Delta semantics:

- return ingested metric rows strictly after the cursor
- frontend appends or de-dupes by `metricId`

### Branches

`listBranches` is a list of branch-anchor spans: one row per anchor-span run,
including nested runs under a different trace root.

Delta semantics:

- return ingested branch rows strictly after the cursor
- frontend appends or de-dupes by branch row identity

Important distinction from traces:

- branches are listable anchor-span rows, not derived trace snapshots
- late child activity under a branch does not re-emit the branch row
- branch delta ordering should therefore use branch-row ingestion order directly

## Backend Data Model Changes

Every pollable stored entity needs explicit ingestion ordering metadata.

Required columns:

- `ingestedAt`

Recommended semantics:

- `ingestedAt`: backend write timestamp, generated as close to insert time as possible
- `ingestedAt` may be nullable during rollout so existing rows do not need backfill
- no new global `ingestId` column by default
- reuse each table's existing stable row ID as the cursor tie-breaker
- keep `ingestedAt` hidden from normal API object payloads

Recommended tie-breakers by endpoint:

- traces activity stream: `dedupeKey`
- branches: `dedupeKey`
- logs: `logId`
- metrics: `metricId`
- scores: `scoreId`
- feedback: `feedbackId`

Important constraint:

- the tie-breaker does not need to be monotonic
- it only needs to be stable and unique within the ordered ingestion stream
- delta mode must only consider rows where `ingestedAt IS NOT NULL`

### ClickHouse v-next

Add `ingestedAt` to:

- trace storage tables needed for polling correctness
- `log_events`
- `score_events`
- `feedback_events`
- `metric_events`
- branch table(s)

Strong recommendation:

- use `Nullable(DateTime64(3, 'UTC'))` for `ingestedAt` in the initial rollout
- do not backfill legacy rows
- populate `ingestedAt` only for newly ingested rows
- reuse the table's existing stable row ID as the cursor tie-breaker
- sort and cursor predicates must use `(ingestedAt, tieBreaker)`

Important trace note:

- trace page ordering may still support presentation fields like `startedAt`
- trace delta correctness must be based on latest trace activity ingestion order
- trace cursors should use the latest `(ingestedAt, dedupeKey)` seen in `span_events`, not `traceId`
- trace delta queries must ignore legacy span rows where `ingestedAt IS NULL`

Rollout consequence:

- page mode continues to work for both legacy and new rows
- delta mode only begins emitting rows once newly written data has non-`NULL` `ingestedAt`
- no historical backfill is required for the first rollout

### DuckDB

Add the same logical columns:

- `ingestedAt`

For current append-only signal tables:

- use nullable `ingestedAt` during rollout
- do not backfill legacy rows
- write `ingestedAt` only for newly inserted rows

For traces:

- both start/end event rows need ingestion metadata
- trace delta queries should compute latest trace activity from all trace span rows
- trace delta tie-breakers should come from the underlying span event row identity, not from `traceId`
- delta queries must ignore rows where `ingestedAt IS NULL`

### In-Memory Store

Add ingestion metadata in-memory for:

- each pollable row record
- each trace entryâ€™s latest activity cursor

This store should mirror backend semantics exactly so tests stay meaningful.

## Query Strategy

### Shared Rule

All delta queries should use:

```sql
WHERE
  ingestedAt IS NOT NULL
  AND (
    ingestedAt > :cursorIngestedAt
    OR (ingestedAt = :cursorIngestedAt AND tieBreaker > :cursorTieBreaker)
  )
ORDER BY ingestedAt ASC, tieBreaker ASC
LIMIT :limitPlusOne
```

Use `limit + 1` to derive `hasMore`.

### Trace Queries

Trace delta queries should:

1. compute latest trace activity cursor per trace
2. filter traces whose latest activity is after the request cursor
3. return current root-trace snapshots for those trace IDs
4. order by latest trace activity cursor ascending for delta mode

Trace page mode can still support display ordering, but should expose a filtered snapshot `liveCursor` for subsequent polling.

The trace activity cursor should be derived from the latest span-event row in the trace:

- primary key: `ingestedAt`
- secondary key: `dedupeKey`
- not `traceId`

## Frontend Merge Contract

Frontend behavior should differ by resource shape.

Trace-like snapshot resources:

- merge by stable identity
- replace prior item with the newest returned snapshot

Append-only row resources:

- append new rows
- de-dupe by stable row ID

This means traces are upsert lists, while logs/scores/feedback/metrics/branches are append lists.

## Validation Rules

Each list endpoint should enforce:

- page mode and delta mode are mutually exclusive
- `after` must be fully specified if present
- `limit` has its own min/max, separate from `perPage`
- delta mode cannot request arbitrary `orderBy`
- if `mode` is omitted, treat the request as page mode
- `mode=delta` may omit `after` to create a "start from now" boundary without backfill
- `after` must not be reused across different filter sets

Recommendation:

- page mode keeps existing allowed orderings
- delta mode always uses ingestion order internally
- page mode returns `liveCursor` for delta resume
- do not introduce a separate `historyCursor` unless we later replace offset pagination
- do not add `ingestedAt` to returned trace/log/metric/score/feedback/branch objects
- do not add `filterHash` to `LiveCursor` in v1
- align `after` query-param transport with existing bracket-notation / JSON-string complex-param handling

## Rollout Plan

### Phase 1: Shared Contract

- add shared `IngestionCursor` type
- add shared delta-mode list argument/response shapes
- update endpoint schemas for traces, logs, scores, feedback, metrics, and branches

### Phase 2: Store Implementations

- ClickHouse v-next
- DuckDB
- in-memory observability store

Implement one signal family at a time:

1. logs, scores, feedback, metrics
2. traces
3. branches

This order is safer because traces have different semantics.

### Phase 3: Server Routing and Client SDK

- extend existing handlers to support delta mode
- update client SDK methods to accept both modes on the same list method
- keep public method names unchanged
- add explicit API docs stating that a `LiveCursor` must be reused with the same filter set it came from

### Phase 4: Frontend Adoption

Switch hot polling paths first:

- traces list
- logs list

Then convert any score/feedback/metric pollers that currently re-fetch full pages.

## Testing Plan

### Shared Contract Tests

- mode validation
- omitted `mode` behaves like page mode
- explicit `mode=page` behaves like default page mode
- `mode=delta` without `after` returns empty items plus a valid `liveCursor`
- `mode=delta` without `after` uses the max matching non-`NULL` ingestion cursor when available
- `mode=delta` without `after` mints a synthetic "now" cursor when no matching non-`NULL` ingestion rows exist
- cursor serialization and parsing
- returned objects do not expose `ingestedAt`
- delta with unchanged filters returns only matching new data
- changing filters requires a new `liveCursor`
- `LiveCursor` shape remains `ingestedAt + tieBreaker` with no `filterHash` in v1
- strict `>` behavior across cursor boundaries

### Store Tests

For every supported store and every supported signal:

- initial page returns `liveCursor`
- delta returns only rows after cursor
- delta ignores rows with `NULL` `ingestedAt`
- same-timestamp collisions do not skip or duplicate rows across polls
- `hasMore` works correctly with `limit + 1`

Extra trace tests:

- late child span re-emits the parent trace
- old `startedAt` / `endedAt` with new ingestion still appears in delta mode
- trace ordering for delta mode follows activity cursor, not display time

### Frontend Tests

- trace upsert behavior by `traceId`
- append behavior for logs/scores/feedback/metrics
- append behavior for branches
- page-mode `liveCursor` resumes delta polling without requiring a second bootstrap request
- no duplicate rows across repeated polls

## Open Questions

- What is the stable branch row identity for frontend de-duplication: `spanId`, `dedupeKey`, or another explicit ID?
- Should `metricId`, `scoreId`, `feedbackId`, and `logId` always be present and stable enough for frontend de-duplication everywhere?
- Should page mode for traces default to `ingestedAt DESC` instead of `startedAt DESC` once this system exists?
- Do we want to expose delta metadata under a dedicated `delta` field, or is `liveCursor + hasMore` enough?
- If offset pagination proves unstable in live views, do we later add a separate `historyCursor` for older-page traversal in display order?

## Strong Recommendations

- extend existing list endpoints instead of adding parallel `/updates` endpoints
- standardize on one public `LiveCursor` shape for every pollable observability list
- use ingestion order for correctness, not event time
- treat traces as activity-driven upsert snapshots, not append-only rows
- include metrics in the first-class design from the start
- treat branches as append-style anchor-span rows, not trace-style activity snapshots
- have page mode return a filtered snapshot `liveCursor` by default so one initial list call can bootstrap live polling
- keep ingestion metadata hidden from normal object payloads and expose it only through cursors
- document clearly that `LiveCursor` is intended to be reused only with the same filter set
