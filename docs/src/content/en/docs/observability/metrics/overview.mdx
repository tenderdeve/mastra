# Metrics overview

Mastra automatically emits performance and usage metrics from traced execution. There's no manual instrumentation needed. Metrics are derived from spans as they complete.

Three categories of metrics are emitted automatically:

- **Duration metrics**: Execution time for agents, workflows, tools, model calls, and processors.
- **Token usage metrics**: Input and output token counts broken down by type (text, cache, audio, image, reasoning).
- **Cost estimation**: Estimated cost per model call based on an embedded pricing registry.

:::note

Metrics require an OLAP-capable store for observability. Relational databases like PostgreSQL, LibSQL, etc. are not supported for metrics. In-memory storage resets on restart.

For local development, use [DuckDB](https://duckdb.org/) through `@mastra/duckdb`. For production, use [ClickHouse](https://clickhouse.com/) through `@mastra/clickhouse`.

:::

## When to use metrics

- Monitor latency across agents, tools, workflows, and model calls
- Track token consumption and cost trends over time
- Identify error-heavy agents or tools by comparing success and error rates
- Compare performance before and after prompt, model, or code changes

## Get started

Install the required packages:

```bash npm2yarn
npm install @mastra/observability @mastra/libsql @mastra/duckdb
```

Then configure observability with a composite store that routes the observability domain to DuckDB:

```ts title="src/mastra/index.ts"
import { Mastra } from '@mastra/core/mastra'
import { LibSQLStore } from '@mastra/libsql'
import { DuckDBStore } from '@mastra/duckdb'
import { MastraCompositeStore } from '@mastra/core/storage'
import { Observability, DefaultExporter, SensitiveDataFilter } from '@mastra/observability'

export const mastra = new Mastra({
  storage: new MastraCompositeStore({
    id: 'composite-storage',
    default: new LibSQLStore({
      id: 'mastra-storage',
      url: 'file:./mastra.db',
    }),
    domains: {
      observability: await new DuckDBStore().getStore('observability'),
    },
  }),
  observability: new Observability({
    configs: {
      default: {
        serviceName: 'mastra',
        exporters: [new DefaultExporter()],
        spanOutputProcessors: [new SensitiveDataFilter()],
      },
    },
  }),
})
```

## Studio

The Studio metrics dashboard visualizes all automatic metrics with KPI cards, detailed breakdowns, and configurable time ranges. See [Studio observability](/docs/studio/observability) for a full walkthrough.

### Filtering the dashboard

The dashboard toolbar supports dimensional filters backed by the underlying spans and metrics:

- **Pick-list filters** (indexed via `LowCardinality` in ClickHouse, cheap on any backend): `rootEntityType`, `entityName`, `serviceName`, `environment`, `tags`.
- **Free-text filters** (backed by `bloom_filter` skip indexes on ClickHouse so point lookups stay fast): `entityId`, `threadId`, `resourceId`, `userId`, `organizationId`, `runId`, `sessionId`, `requestId`, `experimentId`.

All filters are reflected in the URL and can be saved as defaults per browser.

### Memory activity

The dashboard reuses the existing metrics pipeline to report memory activity without requiring any new emitters:

- **Active Threads** â€” count of distinct `threadId` values observed in `mastra_agent_duration_ms`. Uses the new `count_distinct` aggregation (approximate `uniq`/`approx_count_distinct`) so the query stays fast on tens of millions of rows.
- **Active Resources** â€” count of distinct `resourceId` values observed in `mastra_agent_duration_ms`, using the same approximate `count_distinct` aggregation.
- **Memory** card â€” combined view with two tabs:

  - **Top Threads** â€” server-side TopK breakdown of agent runs by `threadId` (with `resourceId`).
  - **Top Resources** â€” server-side TopK breakdown by `resourceId`, aggregating `count_distinct(threadId)` per resource.

  Both tabs use `limit` + `orderBy: 'value'` so only the top 20 rows leave the database.

### Drilldown

Most data points on the dashboard are actionable â€” a click takes you to the **Traces** (primary) or **Logs** (secondary) page pre-filtered to the same window and dimensions so you can see the activity behind the number:

- **Latency** card â€” the header icon opens traces filtered to the active tab's `rootEntityType` (Agent / Workflow / Tool). Clicking a node on the p50 / p95 line narrows the window to the node's hourly bucket.
- **Trace Volume** card â€” the header exposes two icons: open traces scoped to the active tab, and open the logs page filtered to `level=error`. Each bar's "Errors" segment links straight to traces for that entity with `status=error`.
- **Token Usage by Agent** / **Model Usage & Cost** â€” each row links to traces scoped to the agent (`rootEntityType=AGENT` + `entityName`).
- **Scores** card â€” the header opens traces filtered to `rootEntityType=SCORER`; each row in the summary table additionally adds `entityName=<scorer>`.
- **Memory** card â€” Threads rows open traces filtered to the thread id (plus the resource id if known); Resources rows filter to the resource id.

The dashboard's active date range and any dimensional filters are always carried through the drilldown, so you land on the traces / logs page with the same context you had on the dashboard.

## What Mastra measures

Mastra emits three categories of metrics automatically:

- **Duration**: Execution time in milliseconds for agents, workflows, tools, model calls, and processors.
- **Token usage**: Input and output token counts, broken down by type (text, cache, audio, image, reasoning).
- **Cost estimation**: Estimated cost per model call, based on an embedded pricing registry.

Each metric carries trace correlation context so you can drill from a dashboard spike to the exact span that caused it.

For the full list of metric names, labels, and cost fields, see the [Automatic metrics reference](/reference/observability/metrics/automatic-metrics).

## How automatic metrics work

Mastra auto-instruments agent runs, workflow steps, tool calls, and model generations as spans. When a span ends, the observability layer extracts metrics from it:

1. **Duration**: Calculated from the span's start and end timestamps.
1. **Token usage**: Extracted from the `usage` attribute on model generation spans.
1. **Cost estimation**: Runs each token metric through an embedded pricing registry that matches by provider and model name.

Before storage, all metric labels pass through a cardinality filter that blocks known high-cardinality values (such as trace IDs and UUIDs) to keep storage efficient. Metrics are then batched by an internal event buffer and flushed to storage by the `DefaultExporter`.
